## Цель пилота

Прототип контура **ingest → нормализация → чанки → оценка поиска по эталонным вопросам** для экологической и природоохранной документации (в т.ч. нормативка, методики, проектные материалы). Ориентиры по объёму данных и метрикам — в `README.md`.

## Что сделано (результат)

| Направление | Суть работ |
|-------------|------------|
| **Реестр и эталон** | Книга Excel с источниками и золотым набором вопросов (`qa_golden_set`); экспорт в `data/golden_qa.jsonl` для скриптов и будущего API. |
| **Извлечение текста** | Маршрутизация по типу документа: «лёгкий» текстовый слой (PyMuPDF) и «тяжёлый» контур (ранее Docling с OCR; затем — кастомный **DocLayout-YOLO + PyMuPDF + EasyOCR**, предобработка изображений OpenCV). Учтены OOM на больших сканах (батчи, изоляция процессов, даунскейл), кириллица в OCR. |
| **Чанки** | Два режима: по страницам (`data/chunks/`) и **структурные** блоки (`data/chunks_structural/`) с полями типа блока (`block_type`), заголовок секции, уверенность. |
| **Связка с эталоном** | Автоподстановка `chunk_id` в Excel по цитате и странице (внутренние eval-скрипты; в кодовую поставку этапа не входят). |
| **Оценка поиска** | BM25 и dense-поиск по золотому набору — проводились на этапе R&D (скрипты и отчёты в поставку не входят; метрики зафиксированы в этой сводке). |
| **Операторский сервис** | **FastAPI** (`api/main.py`): загрузка файла, ingest, структурные чанки, просмотр в браузере; опционально **Docker** с LibreOffice/Tesseract. Веб-интерфейс: таблицы через **GFM (marked)** + санитизация, доработки по таблицам/формулам/ошибкам сети. |
| **Документация** | `README.md`, `docs/README_INGEST_ROUTER.md`, `docs/api/API.md`, `docs/FAQ.md`, `docs/tz_delivery/`. |

## На чём остановились (текущий статус)

1. **Готово ядро ETL и оценка retrieval** по золотому набору; есть **веб-демо** загрузки и просмотра чанков через API.
2. **Полноценный продуктовый бэкенд** (очереди Celery/Redis, мультипользовательская обработка, SLA) в исходном ТЗ **не реализованы** — это следующий этап после согласования архитектуры.
3. **Генерация ответов (RAG end-to-end)** и метрики groundedness по ответам модели **не внедрены**; есть фундамент: чанки, эталон, поиск.
4. **Таблицы и сложная вёрстка:** простые и двухколоночные таблицы улучшены (извлечение из PyMuPDF, склейка строк, UI); **объединённые ячейки** в смысле HTML **нельзя** честно воспроизвести одной markdown-таблицей — для «пиксельной» верности нужны отдельные решения (HTML, снимок области, другой экспорт).
5. **Качество OCR** на плохих сканах остаётся **риском**: система помечает сомнительные блоки (`needs_review`, `truncated`), но не заменяет ручную вычитку.

## Риски и оговорки для планирования

- Страницы **DOCX** при извлечении через PDF могут **не совпадать** с нумерацией в Word — для эталона по страницам предпочтителен **единый PDF-канон**.
- **Долгие** документы и ответ API: при таймаутах шлюза имеет смысл открывать уже сохранённый документ из базы после фоновой обработки (если такой сценарий будет введён в продукте).
