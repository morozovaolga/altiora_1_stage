# Соответствие задачам ТЗ

## 1. Анализ исходных данных

Что требовалось: исследовать документы заказчика, их форматы, качество и ограничения обработки.

Что сделано:

- описан реестр источников в `inventory_2026-04-17.xlsx`;
- проанализированы форматы `PDF`, `DOC/DOCX`, `RTF`, `TXT`, `XLS/XLSX`, изображения и сканы;
- выделены проблемные случаи: сканы без текстового слоя, старые офисные форматы, таблицы, колонтитулы, плохое качество изображений;
- подготовлена политика обработки входных форматов.

Основные артефакты в поставке:

- `scripts/analyze_source_formats.py`;
- `data/source_format_policy.json` — создаётся при запуске анализа форматов.

Реестр источников (`inventory_*.xlsx`) и корпус документов заказчика в архив поставки **не входят** (см. корневой `README.md`); передаются отдельно по согласованию.

## 2. Выбор модели и подхода

Что требовалось: выбрать open source модель и подход к решению.

Что сделано:

- выбран локальный гибридный подход без внешних API;
- для цифрового текста используется `PyMuPDF`;
- для офисных документов используется конвертация через `LibreOffice`;
- для сканов используются `Tesseract` и `EasyOCR`;
- для подготовки изображений используется `OpenCV`;
- для layout-анализа используется `DocLayout-YOLO`;
- Docling проверен как альтернативный вариант, но не выбран основным маршрутом из-за требований к устойчивости и памяти.

Основные артефакты:

- `scripts/ingest_router.py`, `scripts/ingest_yolo_doctr.py` (основной heavy-маршрут);
- `scripts/ingest_docling.py` — альтернативный маршрут (веб-форма и `parser_mode=docling`);
- `docs/README_INGEST_ROUTER.md`, `docs/FAQ.md`.

## 3. Разработка ETL-пайплайна

Что требовалось: построить пайплайн обработки документов.

Что сделано:

- реализована загрузка документов из реестра и через API;
- добавлена маршрутизация обработки по типу документа;
- реализована конвертация офисных форматов;
- реализованы извлечение текста, OCR, очистка и нормализация;
- реализовано структурное разбиение документов на блоки;
- результаты сохраняются в едином JSONL-формате.

Основные артефакты:

- `scripts/ingest_router.py`, `scripts/ingest_yolo_doctr.py`, `scripts/ingest_docling.py`;
- `scripts/chunk_structural.py`, `scripts/enrich_chunks.py`, `scripts/md_table_postprocess.py`, `scripts/text_sanitize.py`;
- `data/ingested/<SRC>/pages.jsonl`, `data/ingested/<SRC>/manifest.json`;
- `data/chunks_structural/<SRC>.jsonl` — появляются после запуска пайплайна (в «чистой» поставке каталог `data/` может быть пустым).

## 4. Тестирование на данных заказчика

Что требовалось: проверить решение на данных заказчика.

Что сделано:

- пайплайн запускался на смешанном наборе документов из реестра;
- проверялась обработка текстовых PDF, офисных документов, сканов и изображений;
- проверялась устойчивость обработки тяжелых и плохо читаемых документов;
- проводились сравнения альтернативных маршрутов обработки;
- фиксировались статусы обработки и ошибки в `manifest.json`.

Основные артефакты:

- `data/ingested/`, `data/chunks_structural/` — результаты прогонов (при передаче — по согласованию, не обязательны в кодовой поставке);
- `docs/Хронология_для_заказчика.md` — сводка этапов и ограничений для заказчика.

## 5. Разработка API

Что требовалось: разработать API для доступа к пайплайну.

Что сделано:

- реализован сервис на `FastAPI`;
- добавлены сценарии загрузки файла и получения результата обработки;
- добавлены эндпоинты просмотра обработанных документов;
- сервис упакован в Docker;
- в Docker включены системные зависимости для OCR и конвертации офисных документов.

Основные артефакты:

- `api/main.py`;
- `docs/api/API.md`, `docs/FAQ.md`;
- `Dockerfile`;
- `docker-compose.yml`;
- `.dockerignore`.

## Итоговая поставка по ТЗ

Проект закрывает заявленные задачи ТЗ: выполнен анализ входных данных, выбран локальный open source подход, разработан ETL-пайплайн, проведено тестирование на документах заказчика и подготовлен API-прототип для интеграции.
