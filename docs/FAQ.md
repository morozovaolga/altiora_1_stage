Ответы на вопросы по веб-предпросмотру и структуре блоков.

## Что означает плашка рядом с «Документ загружен»?

Это поле **`route`** из ответа API: какой путь обработки был выбран для файла.

| Плашка | Кратко |
|--------|--------|
| **HEAVY** | Обычный **PDF** (или картинка): разметка страницы моделью **DocLayout-YOLO**, текст и таблицы — **PyMuPDF** (при необходимости **EasyOCR**). |
| **DOCLING** | Режим **`parser_mode=docling`**: отдельный пайплайн Docling. На **сложных PDF** часто даёт **более аккуратный** разбор структуры и текста, чем маршрут YOLO+PyMuPDF, но **обычно требует больше RAM** и времени; на слабых машинах или тяжёлых сканах возможны **503** / сбои (см. ниже). |
| **NATIVE_TABLE** | **Excel / CSV** — разбор через **pandas**, без YOLO. |
| **NATIVE_TXT_API** | **TXT** — прямое чтение текста. |
| **NATIVE_OFFICE_API** | **DOCX / RTF** без конвертации в PDF через LibreOffice — упрощённый нативный разбор. |

Для PDF через стандартную форму веб-интерфейса типичен именно **HEAVY** (тяжёлый контур с YOLO), если не выбран **Docling**.

### Режим парсера в веб-форме

Под зоной загрузки файла есть список **«Режим парсера»**: **Авто**, **YOLO + тяжёлый контур**, **Docling**. Значение уходит в поле **`parser_mode`** запроса `POST /extract`.

**Авто и YOLO** при загрузке через сайт для **PDF и изображений** дают **один и тот же** результат: в API всегда включён принудительный тяжёлый маршрут (`force_heavy`), поэтому оба режима уходят в **HEAVY** (YOLO + тот же пайплайн, что и для «явного» YOLO). Практическая разница на веб-форме — в основном **семантическая** (задел под пакетный ingest и единообразие с CLI).

**Docling** — отдельный парсер: на сложных макетах часто **удобнее по качеству** структуры текста, чем YOLO+PyMuPDF, но **заметно требовательнее к оперативной памяти** и времени. Для Excel/TXT по-прежнему используется **нативный** разбор, независимо от выбора в списке.

На больших PDF на CPU у Docling возможны сбои с **нехваткой памяти** (в логах — `std::bad_alloc`, падение subprocess). Сообщение **503** не должно автоматически означать «у вас огромный файл»: при пустом результате без исключения в manifest подставляется отдельный текст; про RAM пишем отдельно, если в `manifest.json` или логах действительно есть `bad_alloc`. Имеет смысл смотреть поле **`error`** в `data/ingested/<source_id>/manifest.json` и лог uvicorn.

Для PDF «лёгкий» только PyMuPDF через эту форму **недоступен**.

## Почему в предпросмотре нет «настоящего» rowspan, как в исходном PDF?

В интерфейсе таблица строится из **markdown** (формат GitHub-таблиц с вертикальными чертами). В таком markdown **нет** способа задать объединение ячеек по вертикали или горизонтали (`rowspan` / `colspan` в смысле HTML).

Разрешение этих атрибутов в настройках **очистки HTML** означает лишь: если когда-либо в чанк попадёт уже готовая HTML-таблица с объединениями, браузер её не «обрежет» по безопасности. **Текущий пайплайн из PDF в основном собирает pipe-markdown**, поэтому сложная шапка с объединёнными ячейками в PDF превращается в **несколько строк** сетки — визуально это не совпадает с оригиналом, хотя данные в ячейках могут быть корректны.

Для «пиксельного» совпадения со сложной вёрсткой таблицы нужны отдельные подходы (например, HTML из специализированного экспорта, фрагмент изображения области, другой формат хранения).

## Таблица разорвана на две страницы: почему одна карточка, а иногда две?

**Склейка двух фрагментов одной таблицы** выполняется на этапе подготовки данных — в скрипте **`chunk_structural.py`**: два подряд блока с типом таблицы и одним документом сливаются в один чанк, если выполняются правила (продолжение без лишнего заголовка, повторяющаяся шапка и т.д.). Тогда в JSON остаётся **один** блок — в UI **одна** карточка **TABLE**.

**В самом браузере** две соседние карточки **TABLE** специально **не склеиваются** в одну: объединение секций в предпросмотре не распространяется на тип «таблица». Поэтому **две плашки TABLE** означают: на шаге структурных чанков слияние **не произошло** (условия не подошли, нужна пересборка данных или доработка правил под ваш кейс), а не отдельную «склейку только в интерфейсе».
