# HTTP API (FastAPI)

Единственный операторский интерфейс к пайплайну: загрузка файла → ingest → структурные чанки.

## Эндпоинты

| Метод | Путь | Описание |
|--------|------|----------|
| `GET` | `/` | HTML-форма загрузки и предпросмотр результатов. |
| `POST` | `/extract` | `multipart/form-data`: поле `file` — документ; опционально **`parser_mode`** — `auto` \| `yolo` \| `docling` (в HTML-форме на `/` — выпадающий список «Режим парсера»). Если не передано, используется переменная окружения **`ALTIORA_DEFAULT_PARSER`** или `auto`. Ответ JSON с чанками. При провале **Docling** (нет текста, ненулевой код subprocess) — **503** с пояснением в `detail`. |
| `GET` | `/documents` | Список `source_id`, для которых есть `data/chunks_structural/<id>.jsonl`. |
| `GET` | `/documents/{source_id}` | JSON с чанками документа. |
| `DELETE` | `/cleanup` | Удаление артефактов с префиксом `API-*` (см. реализацию). |


### `parser_mode`: `auto` и `yolo` в `POST /extract`

Вызов `decide_route` из веб-API передаёт **`force_heavy=True`**. Для **`auto`** сразу после проверки на `docling` срабатывает та же ветка принудительного **heavy**, что и для явного **`yolo`** (оба дают маршрут **HEAVY** до специфичной для PDF логики). В результате для **загрузки через браузер** по типичному **PDF** (и изображениям) **`auto` и `yolo` эквивалентны**: один и тот же YOLO+тяжёлый пайплайн, одно и то же поле **`route`** в ответе. Различие **`yolo`** vs **`auto`** сильнее проявляется в **пакетном** `ingest_router.py` по реестру (где нет принудительного `force_heavy` из API). **`docling`** — отдельная ветка парсера: на сложных документах часто **лучше по качеству** разбора, но **обычно тяжелее по RAM** и времени.

### Ошибки при `parser_mode=docling`

Если дочерний процесс Docling завершился с **ненулевым кодом**, не создан `pages.jsonl` или в нём нет данных, сервер отвечает **`503 Service Unavailable`** с полем `detail`: краткое пояснение (при наличии — текст из `manifest.json` → `error`). Типичные причины: **нехватка RAM** на больших PDF (в логах Docling встречается `std::bad_alloc` в preprocess), аварийное завершение нативного кода. Имеет смысл использовать **`auto`/`yolo`**, разбить документ или увеличить память.

### Прогрев модели при старте API

По умолчанию FastAPI на старте прогревает YOLO-экстрактор (не ждём первый запрос для скачивания/инициализации весов). Отключается переменной окружения **`ALTIORA_PRELOAD_YOLO_ON_STARTUP=0`**.

## Формат чанка в `chunks[]`

Каждый элемент массива `chunks` в ответах `POST /extract` и `GET /documents/{source_id}` — одна строка из structural JSONL: те же поля, что пишет `scripts/chunk_structural.py` (и при необходимости дополняет `scripts/enrich_chunks.py`).

Обязательные для клиента поля зависят от задачи; ниже — **поля качества и структуры**, которые стоит учитывать в UI, фильтрах и приёмочных проверках.

### Поля качества и структуры

| Поле | Тип (логически) | Описание |
|------|-----------------|----------|
| `structure_confidence` | число, 0–1 | Уверенность эвристики **типа блока** (заголовок, абзац, таблица, список и т.д.). Задаётся при классификации фрагмента текста в `chunk_structural.py`. Низкие значения чаще сочетаются с `needs_review: true`. |
| `is_continuation` | boolean | Текст похож на **продолжение предыдущего** блока (начало без заглавной буквы и т.п.). После генерации чанков `chunk_structural.py` выполняет **пост-склейку**: такой чанк вливается в предыдущий, если оба `paragraph`/`list`/`other`, один `doc_id` и суммарная длина ≤ `max_chars` (см. `merge_continuation_into_previous`). |
| `is_truncated` | boolean | Подозрение на **обрезанный** фрагмент (например, неестественный конец строки, дефис в конце без продолжения). Имеет смысл подсвечивать для ручной проверки или дозагрузки страницы. |
| `needs_review` | boolean | Агрегированный флаг **«желательна ручная проверка»**: рассчитывается в `chunk_quality_flags()` по правилам вроде низкой `structure_confidence`, `is_truncated`, подозрительных шаблонов текста, нумерованного пункта, ошибочно принятого за заголовок и т.д. Не заменяет экспертизу, но помогает приоритизировать очередь правок. |

Источник логики: `scripts/chunk_structural.py` — функции `classify_block`, `chunk_quality_flags`, склейка continuation между страницами.

### Связанные поля (кратко)

- `block_type`, `block_level`, `parent_header_id`, `section_title` — структура документа для навигации и RAG.
- `entities` — извлечённые подсказки (ссылки на таблицы/рисунки, измерения и т.п.), при наличии постобработки в `enrich_chunks.py`.
- `formula_vars` — может появиться у блока `formula`, если следующий блок с пояснением «где …» был объединён при обогащении.

## Поле `route` в ответе `POST /extract`

Рядом с заголовком «Документ загружен» в HTML-интерфейсе показывается то же значение, что приходит в JSON как **`route`** (плашка в верхнем регистре).

| Значение `route` | Смысл |
|------------------|--------|
| **`heavy`** | Типичный **PDF** (или изображение): разметка **DocLayout-YOLO** + извлечение текста/таблиц (**PyMuPDF**, **EasyOCR** на сканах). |
| **`docling`** | Выбран режим **`parser_mode=docling`** (в веб-форме — пункт «Docling» или то же поле в `POST /extract`). |
| **`native_table`** | **XLS / XLSX / CSV** — разбор через **pandas**, без YOLO. |
| **`native_txt_api`** | **TXT** — нативное чтение текста. |
| **`native_office_api`** | **DOCX / RTF** без доступного **LibreOffice** для конвертации в PDF — упрощённый нативный разбор без полного heavy-контура. |

См. подраздел **«`parser_mode`: `auto` и `yolo`»** выше. Для Excel и TXT итоговое значение **`route`** перезаписывается на **`native_*`**, как в таблице. Чистый «light»-путь PyMuPDF **без** YOLO для PDF через эту форму **не используется** (см. `api/main.py` и `scripts/ingest_router.py`).

## Таблицы в предпросмотре (HTML) и объединённые ячейки

1. **Формат в UI:** текст чанка с `block_type: table` (или похожий на markdown-таблицу) превращается в HTML через **GFM** (**`marked`**) и проходит **DOMPurify**. В конфигурации санитайзера разрешены атрибуты вроде **`align`**, **`colspan`**, **`rowspan`** — это **запас** на случай готового HTML-таблицы, а не утверждение, что текущий пайплайн **строит** объединённые ячейки.

2. **Markdown и `rowspan` / `colspan`:** стандартные **pipe-таблицы GitHub** не выражают вертикальное или горизонтальное объединение ячеек. Если в исходном PDF шапка нарисована с **`rowspan`** (как в «Таблица 2.2.3» с длинной подписью в первом столбце), извлечение даёт **плоскую сетку** или несколько визуальных строк на одну логическую шапку — в markdown это выглядит как несколько рядов `|…|`, а не как одна объединённая ячейка. Это **ограничение представления**, а не «отключённый rowspan в браузере».

3. **Улучшения на стороне ingest (YOLO + PyMuPDF):** для таблиц используется в первую очередь **`Table.extract(layout=True)`** и ручная сборка pipe-markdown; для **двух колонок** добавлена эвристика склейки строк с пустой левой/правой ячейкой (`scripts/md_table_postprocess.py`). Это **не заменяет** настоящий `rowspan` в сложных шапках.

## Таблица, разорванная между страницами

- **Склейка в данных:** в **`scripts/chunk_structural.py`** (`merge_continuation_into_previous`) два подряд чанка с **`block_type: table`** и одним **`doc_id`** могут быть **слиты в один** чанк: например, если второй фрагмент продолжает markdown без нового заголовка `[Таблица]` и без лишней строки-разделителя, либо если второй чанк повторяет **ту же шапку**, что и первый (общий префикс строк до разделителя `|---|`). Для этого задан отдельный лимит длины склейки таблиц. После слияния в JSONL остаётся **один** чанк — в UI будет **одна** карточка `TABLE`.

- **Предпросмотр в браузере:** функция **`mergeAdjacentSectionUiChunks`** **не объединяет** соседние карточки типа **`table`** и **`formula`** (см. `uiChunkNonMergeableType` в `api/main.py`). То есть **две отдельные плашки `TABLE`** для одной логической таблицы означают, что на этапе **`chunk_structural.py`** слияние **не произошло** (условия не выполнились, устаревший JSONL, разный префикс шапки и т.д.), а не «склейку только в UI».

- **Что делать:** пересобрать **`data/chunks_structural/<doc_id>.jsonl`** после обновления логики `chunk_structural.py` / ingest; при необходимости доработать правила склейки под ваш кейс (вторая страница только из строк данных `|0,7|…|` без повторения шапки).
