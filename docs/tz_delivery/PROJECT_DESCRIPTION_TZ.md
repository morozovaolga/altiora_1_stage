# Описание проекта в рамках ТЗ

## 1. Общая цель

Проект направлен на разработку автономного ETL-пайплайна для извлечения данных из экологической и природоохранной документации. На первом этапе решается задача распознавания текста, разбиения документов на блоки и подготовки результата для дальнейшего использования в прикладных сценариях: поиске по запросу, анализе документов и последующей интеграции с RAG-пайплайнами.

Решение должно работать локально, без обязательного подключения к внешним сервисам и внешним API. Это важно для обработки внутренних документов заказчика и материалов, которые не должны передаваться в сторонние облачные системы.

## 2. Анализ исходных данных

В рамках проекта были исследованы входные форматы и особенности документов заказчика. Корпус включает текстовые, офисные и графические документы:

- `PDF`;
- `DOC` / `DOCX`;
- `RTF`;
- `TXT`;
- `XLS` / `XLSX`;
- изображения и сканы: `JPG`, `JPEG`, `PNG`, `TIFF`, `BMP`.

При анализе выявлены основные технические сложности:

- часть PDF содержит полноценный текстовый слой, который можно извлекать без OCR;
- часть документов представлена сканами или изображениями и требует распознавания текста;
- офисные форматы необходимо приводить к единому виду перед обработкой;
- в документах встречаются таблицы, заголовки, списки, колонтитулы, номера страниц и служебный шум;
- качество сканов неоднородно: возможны перекосы, слабый контраст, старая машинопись и плохо читаемые страницы.

Для управления источниками используется реестр `source_registry` в Excel-книге. В нем фиксируются идентификаторы документов, пути к файлам, форматы и дополнительные признаки, необходимые для запуска обработки.

## 3. Выбор модели и подхода

Для проекта выбран гибридный локальный подход, который сочетает быстрые инструменты извлечения текста, OCR и компьютерное зрение. Такой подход позволяет обрабатывать разные типы документов без обращения к внешним API.

Основные компоненты выбранного подхода:

- `PyMuPDF` - извлечение цифрового текстового слоя из PDF;
- `LibreOffice` - конвертация офисных форматов в PDF перед обработкой;
- `OpenCV` - предобработка изображений и сканов: выравнивание, улучшение контраста, подготовка к OCR;
- `Tesseract` / `EasyOCR` - локальное распознавание текста на сканах и изображениях;
- `DocLayout-YOLO` - анализ структуры страниц и выделение областей документа;
- `Python`, `NumPy`, `Pillow` и вспомогательные библиотеки - реализация обработки и нормализации данных.

В ходе проекта также проверялся Docling как альтернативный Document AI-парсер. Он показал сильные стороны на layout и таблицах, но из-за требований к устойчивости и потреблению памяти основным маршрутом для текущего прототипа выбран кастомный автономный пайплайн на базе локальных компонентов.

## 4. Разработка ETL-пайплайна

Разработан ETL-пайплайн, который принимает документы разных форматов, извлекает из них текст и структуру, очищает результат и сохраняет данные в едином формате.

Основные этапы пайплайна:

1. Загрузка документа из реестра или через API.
2. Определение формата и профиля документа.
3. Конвертация офисных форматов в PDF при необходимости.
4. Извлечение текстового слоя из PDF, если он доступен.
5. OCR-распознавание для сканов и изображений.
6. Предобработка страниц средствами компьютерного зрения.
7. Очистка текста от служебного шума.
8. Разбиение текста на страницы и структурные блоки.
9. Сохранение результата в JSONL-формате.

Единый результат обработки сохраняется в виде:

- `data/ingested/<SRC>/pages.jsonl` - постраничный текст документа;
- `data/ingested/<SRC>/manifest.json` - сведения о статусе обработки, формате, маршруте и ошибках;
- `data/chunks_structural/<SRC>.jsonl` - структурированные блоки текста для последующего использования.

Структурная нарезка документа выделяет заголовки, абзацы, списки, таблицы и прочие фрагменты. Для каждого блока сохраняются признаки качества и структурные поля, позволяющие использовать результат как нормализованный выход ETL-пайплайна.

## 5. Тестирование на данных заказчика

Пайплайн проверялся на документах заказчика из реестра. Тестирование было направлено на подтверждение того, что система:

- корректно открывает и обрабатывает разные форматы файлов;
- извлекает текст из документов с цифровым слоем;
- распознает текст на сканах и изображениях;
- обрабатывает офисные документы через конвертацию;
- сохраняет результат в едином формате;
- не требует подключения к внешним сервисам;
- устойчиво работает в локальном окружении и Docker-контейнере.

Отдельное внимание уделялось устойчивости на тяжелых сканах и документах с плохим качеством исходного изображения. Для этого добавлены предобработка изображений, локальный OCR, контроль качества распознавания и отбраковка нечитаемых фрагментов.

По итогам тестирования подтверждена работоспособность ETL-ядра на смешанном корпусе документов. Результаты обработки пригодны для дальнейшего использования в поиске, анализе документов и последующих RAG-сценариях.

## 6. Разработка API

Для доступа к пайплайну разработан прототип микросервиса на `FastAPI`. API позволяет загружать документы, запускать обработку и получать результат в виде структурированных блоков.

Основные сценарии API:

- загрузка файла пользователем;
- запуск обработки документа;
- получение результата обработки в JSON;
- просмотр списка обработанных документов;
- получение структурных блоков по конкретному документу;
- очистка временных артефактов.

Сервис упакован в Docker-образ. В контейнер включены системные зависимости для OCR, обработки изображений и конвертации офисных документов, включая `tesseract-ocr-rus` и `LibreOffice`. Это упрощает развертывание на сервере заказчика и снижает зависимость от ручной настройки окружения.

## 7. Результат по ТЗ

В результате проекта подготовлен автономный ETL-пайплайн, который:

- поддерживает текстовые, офисные и графические документы;
- извлекает текст из PDF, офисных файлов, сканов и изображений;
- выполняет OCR и предобработку изображений локально;
- приводит результат к единому JSONL-формату;
- разбивает документы на структурные блоки;
- предоставляет API для загрузки и обработки документов;
- может запускаться локально и в Docker;
- пригоден для дальнейшей интеграции с RAG-пайплайнами.

Таким образом, реализована основная цель ТЗ: разработан микросервисный ETL-контур для извлечения данных из документов заказчика и подготовки этих данных к дальнейшему использованию в интеллектуальных сценариях обработки документации.
